# Versioned Relationship Snapshot Publisher

## Purpose

T06 convierte relaciones validadas en una publicacion versionada.

El snapshot es la unidad canonica que consume T07.

T06 receives `ValidatedProductRelationship[]` from T05 and builds an immutable `ProductRelationshipSnapshot`.

T06 no calcula, corrige ni vuelve a validar metricas estadisticas.

## Snapshot

A snapshot is a canonical JSON-ready representation of approved product relationships.

It is not an Excel file, PDF, panel, endpoint, cache, or runtime index. Those can be derived later from the snapshot.

The V1 snapshot contains:

- `schemaVersion`
- `snapshotId`
- `modelVersion`
- `evidenceWindow`
- `relationshipCount`
- `relationships`

It does not include hydrated product names, prices, stock, margin, customer data, publication database IDs, ranking, or runtime indexes.

## Responsibilities

T06 separates three responsibilities:

- Builder: constructs the immutable canonical snapshot.
- Store: persists and activates snapshots through a port.
- Publisher: coordinates `build -> save -> activate`.

The in-memory store is provided for tests, local development, and preparation for T07.

El store en memoria no es persistencia productiva.

## Content Identity

El snapshotId depende del contenido y no del momento de publicacion.

Publicar el mismo contenido produce el mismo snapshotId.

The ID format is:

```text
sha256:<hex>
```

The hash uses:

- `schemaVersion`
- `modelVersion`
- `evidenceWindow`
- canonically ordered `relationships`

It does not use:

- `publishedAt`
- input order
- clock values
- random IDs
- store state
- adapter names

## Canonical JSON

The hash is calculated from an explicit canonical JSON serializer.

Rules:

- object keys are sorted lexicographically;
- arrays keep their canonical array order;
- strings use valid JSON escaping;
- booleans and `null` are preserved;
- non-finite numbers are rejected;
- `undefined`, functions, symbols, BigInt, non-plain objects, and cycles are rejected.

This avoids relying on object construction order from `JSON.stringify(input)`.

## Canonical Order

Relationships are sorted by:

1. source `productId`
2. source `combinationId`, base product first
3. target `productId`
4. target `combinationId`, base product first
5. `relationshipType`
6. `modelVersion`
7. `evidenceWindow.from`
8. `evidenceWindow.to`

The snapshot does not sort by `reliability`, `confidence`, `lift`, `jointCount`, or input order.

Ranking belongs to a later runtime or commercial policy layer.

## Homogeneity

A snapshot is one publication unit. All relationships in a non-empty snapshot must share:

- `modelVersion`
- `evidenceWindow.from`
- `evidenceWindow.to`

Mixed batches fail with explicit errors. T06 does not split a mixed batch into multiple snapshots.

## Empty Snapshots

Empty snapshots are rejected by default.

They can be built only when `allowEmptySnapshot` is true and explicit metadata is provided:

- `modelVersion`
- `evidenceWindow`

An explicitly empty snapshot emits `EMPTY_SNAPSHOT_PUBLISHED`.

This supports controlled deactivation without allowing accidental empty publications.

## Immutability

The builder deep-clones relationship content before building the snapshot and freezes the result recursively.

Input wrappers and relationships are not frozen or modified.

Mutating the original input after build does not change the snapshot.

## Store Semantics

The `ProductRelationshipSnapshotStore` port supports:

- `save`
- `activate`
- `getById`
- `getActive`

`save` is idempotent by `snapshotId`.

Saving the same canonical snapshot twice returns `already_exists`.

Saving different content under an existing ID fails with `SNAPSHOT_ID_COLLISION`.

Saving does not activate. Activation is explicit.

Only one snapshot can be active at a time, and older snapshots remain available by ID.

## Publisher

The publisher performs:

```text
build
save
activate
```

It returns the snapshot, save status, activation flag, statistics, warnings, and optional `publishedAt`.

`publishedAt` is copied from `ProductRelationshipSnapshotPublicationContext`. It is not generated by the domain and does not participate in `snapshotId`.

## Atomicity

The domain publisher preserves logical order: save before activate.

Los adaptadores productivos deberan implementar save + activate con garantias transaccionales o con un mecanismo de activacion atomica.

The in-memory store behaves atomically for tests. T06 does not simulate a distributed transaction.

## Compatibility

T06 consumes T05 validated relationships.

It preserves the T01B `CalculatedProductRelationship` contract and does not depend on T02 readers or datasets.

It does not recalculate T03 evidence.

It does not recalculate T04 reliability.

It does not re-apply T05 reliability thresholds or association policies.

## Excluded Scope

T06 does not implement SQL, migrations, Redis, Neo4j, endpoints, cron jobs, runtime readers, Excel export, PDF export, web panels, CRM integration, e-commerce integration, product hydration, stock filters, margin filters, commercial ranking, personalization, or new statistical formulas.
